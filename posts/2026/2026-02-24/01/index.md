# 이미지 검증 배치

# 들어가기 전 

호텔 공급사 데이터 적재 이후 우리는 호텔 및 룸 이미지를 검증해야 했다.
표면적으로는 단순하다. URL을 호출하고, 이미지를 디코딩하고, width와 height를 추출하고, 우리가 정의한 기준에 따라 quality_score를 계산한 뒤 상태를 갱신하면 된다. 

<br> 

그러나 대상이 1억 건을 넘어가는 순간, 이 문제는 이미지 검증 문제가 아니다. 이 문제는 비용 분배(cost allocation) 문제다.

<br> 


이미지 한 장의 평균 크기를 300KB로 가정하면 전체 트래픽은 30TB에 이른다. 이것은 단순 총량 문제가 아니다. 이 트래픽은 동시성 점유 시간으로 환산되어 시스템을 압박한다. 네트워크는 병목이고, CDN은 외부 통제 영역이며, TCP 연결 수는 유한하다. 이 조건에서 “모든 이미지를 끝까지 다운로드하여 정확히 판정하자”는 접근은 이론적으로는 맞지만 실무적으로는 붕괴한다. 

# 제목 추천 

전체 이미지 갯수는 1억개 이상. 게다가 처리해야할 이미지는 앞으로 계속해서 더 늘어날 가능성이 농후하다. 
그래서 가장 첫번째로는 논리적으로 이 이미지들을 그룹핑을 시켰다. 
각각 공급사별로, 스코프로(호텔인지, 룸인지) 나눴다. 
이때 당시 처리하던 공급사는 4개. 물론 공급사마다 제공하는 이미지 갯수는 달랐지만 그냥 산술적으로 계산해버리면 약 1250만건이 하나의 잡에서 처리할 총 이미지 갯수가 된다. 

이제 이미지 검증 배치는 이제 공급사, 스코프에 대한 잡 파라미터로 관리가 될 것이다. 스케줄러에 등록하고 총 8번을 돌아야 끝나는 무거운 작업이지만 일단 나아가보자. 

초기 구현은 다음과 같았다. 

URL 호출 → 전체 다운로드 → 디코딩 → pHash 생성 → 판정 → update

로컬 테스트 환경에서는 아무 문제가 없었다. 수천 건 단위에서는 충분히 빠르게 동작했다. 그러나 실제 배치 로그를 확인했을 때 병목은 명확하게 드러났다.

validateMs가 4500ms 수준에서 수렴하고 있었다. 평균 probe 시간은 4.5초였다. 동시성 200으로 실행하면 이론적으로는 200개의 요청이 병렬로 수행되지만, 실제로는 200개의 소켓이 3.5 ~ 4.5초씩 점유된다. 느린 CDN 하나가 해당 동시성 슬롯을 장기간 점유하면, 전체 처리량은 급격히 감소한다. 

여기서 중요한 것은 Virtual Thread가 이 문제를 해결하지 못한다는 점이다. Virtual Thread는 carrier thread를 점유하지 않을 뿐, 네트워크 자원을 무한히 생성해주지 않는다. 병목은 CPU가 아니라 외부 I/O다.

CPU 측면도 마찬가지다. pHash 생성은 단순 width/height 추출과는 차원이 다른 비용을 가진다. 1억 건 × 10ms만 계산해도 100만 초의 CPU 시간이 필요하다. 이것은 수평 확장 없이는 현실적으로 감당 불가능하다.

이 시점에서 설계를 바꿔야 했다.

# 단계적 배분

모든 이미지를 동일한 비용으로 검사하지 않는다.
이미지 검증을 “정확도 단계”가 아니라 “비용 단계”로 분해한다. 
따라서 기존 하나의 잡을 두개로 구성한다. 

## 첫번째 스텝: FAST PROBE

첫 번째 단계의 목적은 정확한 판정이 아니다. 대상을 최대한 빠르게 축소하는 것이다. 

connect/read timeout을 공격적으로 낮춘다. Range 요청을 사용해 전체 바이트가 아니라 일부만 수신한다. Content-Type만 확인한다. 가능하다면 헤더 기반으로 width/height를 추출한다.
이 단계에서 우리는 대부분의 이미지를 valid 또는 invalid로 수렴시킨다. 전체 다운로드는 수행하지 않는다. 디코딩도 수행하지 않는다.

실제 운영 로그에서 FAST PROBE 도입 이후 평균 시간은 정확히 내가 설정한 시간 값만큼으로 떨어졌다.
그리고 전체 대상의 꽤 유의미한 수치만큼 이 단계에서 종료되었다.
물론 정확도는 약간 희생된다. 그러나 처리량은 기하급수적으로 개선된다. 

## 두번째 스텝: NORMAL PROBE

FAST 단계에서 애매한 케이스들은 첫번째 스텝에서 특정 상태값으로 저장한다. 예를 들어 가장 간단한 `error` 라든지 두 번째 스탭에서 처리할 수 있게 상태 값을 업데이트한다.
헤더 파싱 실패, MIME 불일치, 일시적 네트워크 오류 등이 여기에 해당한다. 

이 단계에서는 timeout을 완화하고, 제한적 전체 다운로드를 허용한다. 그러나 여전히 무제한 재시도는 금지한다. 여기서의 목적은 정확도 보정이다. 그러나 대상은 전체의 소수다. 

이 계층 구조가 없다면, 전체 시스템은 항상 최악의 비용 경로를 따라가게 된다.


## pHash 정합성 



처리량을 높이기 위해 Range 요청을 도입하면 전체 바이트를 다운로드하지 않고 일부만 수신할 수 있다. 헤더 기반으로 width/height를 추출할 수 있고, MIME 타입을 확인할 수 있다. 네트워크 비용은 크게 감소한다.

그러나 여기서 즉시 문제가 발생한다. Range 요청으로 일부 바이트만 가져오면, 그 데이터로 계산한 pHash는 full GET 기반 pHash와 동일할 수 없다. pHash는 최종 픽셀 행렬에 대한 DCT 변환 결과이므로, 이미지 전체를 디코딩하지 않으면 안정적인 해시를 보장할 수 없다. 이건 구현상의 한계가 아니라 정보 이론적 한계다. 입력 데이터가 다르면 해시도 달라진다. 

정확히 말하면 동일할 수 없다.

지피티에 물어보면 pHash 값은 `최종 픽셀 행렬을 기반으로 DCT 변환을 수행한 뒤, 저주파 성분을 양자화하여 생성되는 값` 이라고 나온는데 우리가 일반적으로 알고 있는 해시값 즉, 입력값이 다르면 복원된 값이 다르고 그 결과 해시값이 달라지는 그런 개념이다. 


따라서 Range 기반 설계와 full pHash 정확성을 동시에 요구하는 것은 모순이다. 이 둘은 동일한 계층에 둘 수 없다.
이런 개념으로 미루어보면 “Range 기반 pHash”와 “Full GET 기반 pHash”의 정합성을 완전히 동일하게 유지하겠다는 전제는 성립하지 않는다는 결론에 다다른다. 

하지만 이 배치의 목적은 중복 이미지 탐지가 아닌 명백히 문제가 있는 이미지를 골라내는 배치이며 현 상황에서 배치의 목적은 고속 수렴이다. 

pHash를 “절대적 식별자”로 사용하는 것이 아니라, “추가 비용을 들일 가치가 있는 대상인지 판단하는 보조 신호”로 사용한다. FAST PROBE에서 생성된 해시는 완전한 진실을 말해주지 않는다. 대신, 이 대상이 명백히 깨졌는지, 최소 해상도 기준을 충족하는지, 추가 정밀 검사가 필요한지 여부를 판단하는 신호로 기능한다.

따라서 여기서의 전략은 명확하다.

FAST 단계에서 생성된 해시는 “잠정적 해시”다.
정밀 해시가 필요한 경우에만 NORMAL 단계 이후에서 Full GET 기반 해시를 생성한다.
그리고 이 두 해시를 동일한 컬럼에 덮어쓰지 않는다.

우리는 해시를 절대적 식별자로 저장하는 것이 아니라, “검증 레벨에 따라 생성된 메타데이터”로 저장한다. 해시의 계층을 구분하는 순간, 정합성 문제는 모순이 아니라 설계 의도에 부합하는 특성으로 변한다.


# 다양한 이슈

계층형 비용 분리 구조는 처음부터 도출된 결론이 아니었다. 
사실 모든 일이 그렇듯이 처리하고 보면 별일 아니지만 여러가지 현상에 대해 원인을 파악하고 분석하여 결론에 도다라는 길은 험난하기만하다. 
이 구조 또한 여러 번의 병목, 오판, 그리고 인프라 레벨의 제약을 몸으로 겪은 뒤에야 도달한 구조였다.

아래 내용은 FAST/NORMAL 트랙으로 정리되기 이전에 겪었던 문제들이다.
이 경험들이 누적되면서 “모든 이미지를 동일 비용으로 처리하는 설계는 불가능하다”는 확신으로 이어졌다.

## 1. 청크 기반 배치에서 발생한 지연 전파 문제

Spring Batch의 chunk-oriented step은 데이터 처리 관점에서 매우 합리적인 모델이다. 일정 개수의 아이템을 읽고(read), 가공(process)하고, 저장(write)한 뒤 하나의 트랜잭션으로 커밋한다. CRUD 중심 배치에서는 안정적이고 예측 가능한 구조다.


문제는 이 모델이 “외부 네트워크 I/O”를 포함하는 순간 발생한다.

예를 들어 chunkSize = 100이라고 가정하자. 평균 이미지 처리 시간이 300ms라면, 순차 처리 시 약 30초가 소요된다. 이를 개선하기 위해 청크 내부에 비동기 처리나 Virtual Thread를 도입하면, 100개의 요청을 병렬로 던지고 대략 “가장 느린 하나의 시간”에 수렴시킬 수 있다. 평균 지연은 확실히 줄어든다.

그러나 여기서 간과했던 점이 있다.

청크는 단순한 논리적 묶음이 아니라, 트랜잭션 경계라는 사실이다.

99개의 이미지는 200~300ms 내에 완료되었는데, 단 하나의 이미지가 CDN 지연으로 5초 이상 응답하지 않는다면 어떻게 될까. 청크는 그 하나를 기다린다. 비동기 처리를 하더라도 commit은 모든 Future가 완료된 이후에 이루어진다. 결국 청크 단위 트랜잭션은 “가장 느린 하나”의 응답 시간에 의해 결정된다.

이 구조에서는 평균이 의미가 없다.
tail latency가 곧 청크 처리 시간이다.

더 큰 문제는 이 지연이 단순한 처리 지연이 아니라 전파되는 지연이라는 점이다.

청크가 길어질수록 다음과 같은 현상이 발생한다.

- 트랜잭션 점유 시간이 길어지며 DB connection이 장시간 점유된다.
- commit 지연으로 undo/redo 로그 처리량이 불안정해진다.
- 병렬 워커 환경에서는 느린 청크가 하나의 처리 슬롯을 장시간 점유한다.


처음에는 timeout을 낮추는 방식으로 대응했다. 5초 기다릴 것을 1초만 기다리도록 만들었다. 그러나 이는 근본 해결이 아니었다. 네트워크 I/O는 여전히 트랜잭션 내부에 있었고, 청크는 여전히 가장 느린 요청을 기준으로 완료되었다.

여기서 질문을 다시 던졌다.

우리가 보장해야 하는 원자성은 무엇인가?

이미지 URL 호출과 디코딩 과정은 DB 정합성과 직접적인 관계가 없다. 우리가 보장해야 하는 것은 “이미지 상태 업데이트의 일관성”이지, “HTTP 호출과 DB 업데이트가 하나의 원자 연산”일 필요는 없다.

이 인식이 구조 전환의 출발점이었다.

청크 모델은 데이터 정합성 중심의 트랜잭션 모델이다.
그러나 이미지 검증 배치는 외부 I/O 중심의 비용 모델을 가진다.
이 둘은 자연스럽게 충돌한다.

그래서 우리는 청크 모델을 부분적으로 해체하고, claim 기반 tasklet 구조로 재구성했다.

기존 구조는 다음과 같았다.

<br> 


```kotlin
@Bean
fun imageStep(): Step =
    stepBuilderFactory.get("imageStep")
        .chunk<Image, ImageResult>(100)
        .reader(reader())
        .processor { image ->
            // HTTP 호출이 트랜잭션 안에 포함된다
            probeClient.call(image.url)
        }
        .writer(writer())
        .build()
```

<br> 


이 구조에서는 processor 안의 모든 HTTP 호출이 청크 트랜잭션에 묶인다.

이를 다음과 같이 변경했다.

<br> 

```kotlin
override fun execute(contribution: StepContribution, chunkContext: ChunkContext): RepeatStatus {

    // 1) 짧은 트랜잭션 - 대상 선점
    val targets = Tx.run { claimTargets() }

    // 2) 트랜잭션 없음 - 네트워크 I/O 수행
    val results = targets.map { target ->
        probeClient.fastProbe(target.url)
    }

    // 3) 짧은 트랜잭션 - 결과 반영
    Tx.run { batchUpdate(results) }

    return RepeatStatus.CONTINUABLE
}
```

<br> 

트랜잭션 범위를 “claim”과 “update”에만 한정했다.
HTTP 호출은 두 트랜잭션 사이에서 수행된다.

이 구조로 전환한 이후, 느린 이미지 하나는 여전히 느리지만, 최소한 DB 트랜잭션과 커넥션을 장시간 점유하지 않게 되었다. tail latency는 네트워크 계층에 머무르고 데이터 계층으로 전파되지 않는다.

이 차이는 단순한 튜닝이 아니다. 실행 모델 자체를 바꾼 것이다.

### AsyncItemProcessor / AsyncItemWriter는 왜 선택하지 않았는가? 

당연히 AsyncItemProcessor와 AsyncItemWriter도 검토했다.
이 조합은 processor 단계를 병렬화해 평균 처리 시간을 크게 줄일 수 있다.

그러나 이 방식은 청크 트랜잭션 모델 자체를 바꾸지 않는다.

AsyncItemWriter는 내부적으로 Future를 모두 기다린 뒤 commit을 수행한다. 즉, 비동기로 처리하더라도 트랜잭션은 여전히 가장 느린 요청을 기다린다. 병렬화는 가능하지만, 지연 전파 경로는 그대로 남는다.

우리가 해결해야 했던 문제는 단순한 처리 속도가 아니라, 외부 I/O가 트랜잭션 경계를 점유하는 구조였다.

Async는 병렬화 전략이다.
그러나 우리가 필요로 했던 것은 트랜잭션 경계의 재정의였다.

그래서 우리는 병렬성을 추가하는 대신,
지연이 전파되는 경로 자체를 차단하는 방향으로 설계를 전환했다.

## 2. Virtual Thread는 만능이 아니다

초기에는 Virtual Thread를 대량으로 생성하면 처리량이 선형적으로 증가할 것이라 기대했다. 실제로 blocking I/O에서는 carrier thread가 park되므로, 이론적으로는 매우 많은 동시성을 확보할 수 있다.

그러나 운영 환경은 JVM 내부가 아니라 네트워크 바깥에서 병목이 발생한다.

동시성 200 → 400 → 800으로 올려보았다.
처리량은 일정 수준 이상에서 증가하지 않았다. 오히려 변동폭이 커졌다.

이유는 명확했다.

- TCP 소켓 수는 유한하다.
- OS의 ephemeral port는 제한적이다.
- CDN은 rate limit을 가진다.
- 네트워크 대역폭은 공유 자원이다.

Virtual Thread는 JVM 내부 스레드 자원 문제를 해결할 뿐이다. 외부 자원은 여전히 물리적 제약을 가진다.

또한 우리는 CPU 연산과 I/O 연산을 동일 풀에서 처리하고 있었다. 이는 구조적으로 잘못된 선택이었다. 네트워크 I/O 대기와 pHash 계산이 동일 스케줄링 큐에서 경쟁하고 있었기 때문이다.

결국 다음과 같이 분리했다.

I/O 전용 executor (동시성 제한 + 도메인 세마포어)

CPU 전용 executor (availableProcessors 기반 고정 풀)

이 분리 이후, 처리량의 안정성이 눈에 띄게 개선되었다. 중요한 것은 스레드 수가 아니라, “자원 성격에 맞는 격리”였다.


## 3. 하드웨어와 네트워크는 설계의 일부다


대부분의 개발자는 성능 문제를 만나면 먼저 코드와 스레드 수를 조정한다. 나 역시 그랬다. Virtual Thread를 도입했고, 동시성을 늘렸고, timeout을 조정했다. JVM 내부에서 할 수 있는 최적화는 거의 다 건드렸다고 생각했다.

그러나 어느 날, 전혀 다른 방향에서 메시지가 들어왔다.

“혹시 네트워크 부하 걸릴만한 작업 진행 중이실까요? 공유기 커넥션이 크게 증가하고 있습니다.”

그 순간, 시스템의 경계가 JVM 밖으로 확장되었다.

애플리케이션 관점에서 보면 우리는 단순히 HTTP 호출을 병렬로 수행하고 있었을 뿐이다. 그러나 인프라 관점에서 보면 우리는 대량 외부 세션을 생성하는 네트워크 트래픽 발생 장치였다. 실제로 동시 커넥션 수는 순간적으로 3000까지 치솟았다. 이것은 단순한 로그 수치가 아니라, 공유기 세션 테이블을 압박하는 수치였다.

그때 깨달았다.

처리량은 JVM 내부 스레드 수로만 결정되지 않는다.

우리가 늘리고 있던 것은 “동시 요청 수”였지만, 인프라가 감당해야 했던 것은 다음과 같은 물리적 자원들이었다.

그 중 어느 하나라도 임계점에 도달하면, 애플리케이션 레벨의 스레드 증가는 오히려 전체 시스템을 불안정하게 만든다.

특히 네트워크는 우리가 통제할 수 없는 외부 자원이다. CDN의 응답 속도는 시간대에 따라 변하고, 특정 공급사의 트래픽 정책에 따라 지연이 급증할 수 있다. 그럼에도 불구하고 우리는 내부 스레드를 무작정 늘려 외부 시스템을 동시에 두드리고 있었다.

이 구조는 위험하다.

동시성을 늘리는 것은 “처리량을 높이는 행위”가 아니라, “시스템 전체에 압력을 가하는 행위”다.

이 사건 이후 우리는 설계를 다시 바라보았다.
더 이상 스레드를 늘리는 방식으로 문제를 해결하지 않기로 했다.

대신 다음을 명시적으로 설계했다.

첫째, 전역 QPS 제한을 도입했다.
이미지 검증 배치 전체가 초당 몇 개의 외부 호출을 수행할 수 있는지 상한을 설정했다. 처리량은 최대치가 아니라 안정 구간에서 유지되도록 했다.

둘째, 도메인별 동시성 제한을 적용했다.
특정 공급사의 CDN이 느려지면, 해당 도메인의 세마포어가 자연스럽게 병목을 형성하도록 했다. 다른 도메인은 영향을 받지 않는다. 빠른 공급사는 계속 빠르게 처리된다.

셋째, exponential backoff를 명시적으로 도입했다.
429나 5xx 응답은 단순 재시도가 아니라, 지수적 지연을 두고 재시도한다. 외부 시스템을 압박하는 방식이 아니라, 존중하는 방식으로 호출한다.


계속 집착해오던 “최대 처리량” 대신 “지속 가능한 최대 처리량”을 추구하기 위한 인식의 전환이였다.
